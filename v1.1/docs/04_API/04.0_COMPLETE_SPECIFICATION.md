## Executive Summary

**Suburbmates v1.1** is a Melbourne hyper-local marketplace MVP with the following characteristics:

- **Model:** Directory-hybrid (users start as Business Owners, optional Vendor upgrade)
- **Tech Stack:** Next.js 14 + Supabase PostgreSQL + Stripe + Claude Haiku + Vercel
- **Revenue:** 8% commission (Basic tier) / 6% (Pro tier), featured slots, future subscriptions
- **Timeline:** Milestone-based (no fixed launch date); robustness and correctness take priority
- **Support Model:** Chatbot-heavy (70% auto-resolution), no SLAs, <1 hr/week founder load
- **Reuse from Phase 5:** Refunds (85% reusable), disputes (80% reusable), Stripe integration (100%)

**Key Decision:** Hybrid approach = adopt v1.1 architecture (Supabase + Next.js) while preserving Phase 5 features (refunds, disputes) from current project.

## Business Model

### User Types

1. **Business Owner** (FREE entry level)
   - Creates free profile in directory
   - Can claim existing business (ABN verification optional)
   - Sees refund requests from buyers
   - Can upgrade to Vendor

2. **Vendor** (PAID tier)
   - Creates products with prices
   - Connects Stripe account
   - Receives commission-split payouts
   - Can manage refund approvals
   - Can be escalated to disputes

3. **Buyer** (IMPLIED)
   - Searches and browses businesses
   - Purchases products
   - Can request refunds
   - Can escalate disputes

4. **Admin** (FOUNDER)
   - Resolves disputes
   - Manages platform
   - Handles escalations

### Revenue Streams

1. **Commission** (Primary)
   - Basic Tier: 8% + Stripe 2.9% + $0.30 = ~11% total
   - Pro Tier: 6% + Stripe 2.9% + $0.30 = ~9% total
   - Vendor gets: 89-91% of order value

2. **Featured Slots** (Future Phase 2)
   - Prominence in search results ($X/month per vendor)

3. **Subscription Tiers** (Future Phase 2)
   - Basic: 8% commission, 10 products, 5GB storage
   - Pro: 6% commission, unlimited products, 10GB storage

### Unit Economics

- **Breakeven:** ~300 orders/month (~A$1,500 revenue)
- **Sustainable:** ~600 orders/month (~A$3,000 revenue)
- **Margin per order:** A$4.80â€“$6.00 (after all fees)

### Go-to-Market

- Month 1: Beta with 10-20 vendors, 100+ business profiles
- Month 2: Public launch, 50+ vendors
- Month 3: Validation gate (revenue >A$500)
- Month 6: Scale target (300+ vendors, A$3K/month)

## Database Schema

### Core Tables (11 total)

#### 1. profiles (from Supabase auth)
```sql
id          BIGINT PRIMARY KEY REFERENCES auth.users(id)
email       TEXT UNIQUE NOT NULL
name        TEXT
role        TEXT CHECK (role IN ('buyer', 'business_owner', 'vendor', 'admin'))
created_at  TIMESTAMP DEFAULT NOW()
updated_at  TIMESTAMP DEFAULT NOW()
```

#### 2. business_profiles
```sql
id                  BIGSERIAL PRIMARY KEY
owner_id            BIGINT NOT NULL REFERENCES profiles(id) ON DELETE CASCADE
name                TEXT NOT NULL
description         TEXT
suburb              TEXT NOT NULL  -- Melbourne suburb
category            TEXT
phone               TEXT
website             TEXT
verified_abn        TEXT UNIQUE (optional)
profile_image_url   TEXT
created_at          TIMESTAMP DEFAULT NOW()
updated_at          TIMESTAMP DEFAULT NOW()
UNIQUE(owner_id)  -- One business per owner
```

#### 3. vendors
```sql
id                    BIGSERIAL PRIMARY KEY
business_id           BIGINT NOT NULL UNIQUE REFERENCES business_profiles(id)
stripe_account_id     TEXT UNIQUE
stripe_connect_id     TEXT
subscription_status   TEXT DEFAULT 'free' -- free|active|cancelled
total_earnings_cents  BIGINT DEFAULT 0
bank_account_verified BOOLEAN DEFAULT FALSE
created_at            TIMESTAMP DEFAULT NOW()
updated_at            TIMESTAMP DEFAULT NOW()
```

#### 4. products
```sql
id                  BIGSERIAL PRIMARY KEY
vendor_id           BIGINT NOT NULL REFERENCES vendors(id) ON DELETE CASCADE
title               TEXT NOT NULL
description         TEXT
price_cents         BIGINT NOT NULL
category            TEXT
kind                TEXT DEFAULT 'service' -- service|product|package
fulfillment_method  TEXT DEFAULT 'both' -- pickup|delivery|both
stock_quantity      INT DEFAULT 999
image_url           TEXT
is_active           BOOLEAN DEFAULT TRUE
stripe_product_id   TEXT
created_at          TIMESTAMP DEFAULT NOW()
updated_at          TIMESTAMP DEFAULT NOW()
```

#### 5. orders
```sql
id                      BIGSERIAL PRIMARY KEY
buyer_id                BIGINT NOT NULL REFERENCES profiles(id) ON DELETE CASCADE
vendor_id               BIGINT NOT NULL REFERENCES profiles(id)
product_id              BIGINT NOT NULL REFERENCES products(id)
quantity                INT DEFAULT 1
total_cents             BIGINT NOT NULL
payment_status          TEXT DEFAULT 'pending' -- pending|succeeded|failed
fulfillment_status      TEXT DEFAULT 'pending' -- pending|ready|completed|cancelled
stripe_payment_intent_id TEXT UNIQUE
shipping_address        JSONB
notes                   TEXT
created_at              TIMESTAMP DEFAULT NOW()
updated_at              TIMESTAMP DEFAULT NOW()
```

#### 6. refund_requests
```sql
id                 BIGSERIAL PRIMARY KEY
order_id           BIGINT NOT NULL REFERENCES orders(id) ON DELETE CASCADE
buyer_id           BIGINT NOT NULL REFERENCES profiles(id)
reason             TEXT NOT NULL -- not_as_described|changed_mind|defective|other
description        TEXT NOT NULL
status             TEXT DEFAULT 'pending' -- pending|approved|rejected|processing|completed
refund_amount_cents BIGINT
stripe_refund_id   TEXT
vendor_response    TEXT
responded_at       TIMESTAMP
processed_at       TIMESTAMP
created_at         TIMESTAMP DEFAULT NOW()
updated_at         TIMESTAMP DEFAULT NOW()
```

#### 7. disputes
```sql
id                  BIGSERIAL PRIMARY KEY
order_id            BIGINT NOT NULL REFERENCES orders(id)
buyer_id            BIGINT NOT NULL REFERENCES profiles(id)
vendor_id           BIGINT NOT NULL REFERENCES profiles(id)
refund_request_id   BIGINT REFERENCES refund_requests(id)
status              TEXT DEFAULT 'open' -- open|under_review|resolved|escalated
reason              TEXT NOT NULL
buyer_evidence      JSONB -- Array of file URLs
vendor_response     TEXT
admin_decision      TEXT
resolution_status   TEXT -- buyer_refund|vendor_keeps|split
decided_at          TIMESTAMP
decided_by          BIGINT REFERENCES profiles(id)
created_at          TIMESTAMP DEFAULT NOW()
updated_at          TIMESTAMP DEFAULT NOW()
```

```sql
-- FAQs indexed (CSV or JSON, not DB table)
```

## Support Model

### Philosophy

**No SLAs in MVP.** Reliability prioritized; market comparables (Gumtree, eBay, Facebook) don't promise SLAs.

**Chatbot-Heavy Support** = 70% auto-resolution via Claude RAG + Stripe/Sentry read-only lookups

**Founder Load:** <1 hr/week (exceptions + critical issues only)

### Three Support Intents

#### 1. FAQ Chatbot (Claude RAG)
- **Use case:** "How do I refund an order?"
- **Bot action:** Search FAQ docs, synthesize answer via Claude
- **Resolution:** ~60-70% of questions
- **Cost:** ~$2/month
- **Guardrail:** Read-only; never modifies account

#### 2. Billing Auto-Response (Stripe read-only)
- **Use case:** "What was my last charge?"
- **Bot action:** Query Stripe for customer invoices/charges
- **Resolution:** ~90% of billing questions
- **Cost:** Free (within Stripe quota)
- **Guardrail:** Read-only API token; never initiates refunds

#### 3. Technical Auto-Response (Sentry read-only)
- **Use case:** "I'm getting an error on checkout"
- **Bot action:** Search Sentry for matching error + provide self-help
- **Resolution:** ~60-70% of technical questions
- **Cost:** Free (within Sentry quota)
- **Guardrail:** Read-only token; never modifies issues

### Target Metrics

- **Deflection rate:** â‰¥70% (% handled by chatbot, no escalation)
- **Founder load:** <1 hr/week
- **Cost:** ~$2-3/month (Claude only; Stripe/Sentry read-only free)

## Key Decisions

### 1. âœ… Hybrid Approach (Updated)

- Keep Supabase + Next.js as the canonical v1.1 architecture.
- Allow **selective hybrid reuse** from Phase 5 where:
  - The domain model is identical,
  - The code can be safely refactored to match v1.1 naming, schema, and Stripe flows,
  - There are no hidden dependencies that drag in old assumptions.
- **Refunds, disputes, and risk logic are always re-implemented** on top of the v1.1 schema and event flows.
  - Phase 5 code may be read for ideas, but not dropped in directly.
- Hybrid is used to improve **quality and velocity**, not to meet a fixed calendar date.
  - There is **no December 1 deadline**; go/no-go is driven by quality gates, not a hard launch day.

### 2. âœ… Chatbot-Heavy Support (No SLAs)
- Reduces founder ops to <1 hr/week
- Market-validated (Gumtree, eBay, Facebook)
- Enables sustainable solo founder operation

### 3. âœ… Defer Vendor Tiers to Phase 1.5
- MVP = free entry + optional paid features
- Simplifies launch scope
- Implement tiers in January once metrics known

### 4. âœ… Read-Only Stripe/Sentry Integration
- Support chatbot never initiates refunds
- Never modifies Sentry issues
- Reduces legal liability + security risk

### 5. âœ… PostgreSQL + RLS Over MySQL
- Supabase RLS enforces security at DB level
- Simpler than app-level permission checks
- Better for serverless (Vercel Functions)
